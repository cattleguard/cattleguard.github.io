Jenkins, My First Pipeline

Trials and Tribulation

Published Wed, May 27, 2020 by Anthony Towry

*Photo by [Lhu Shi Hui](https://unsplash.com/@lhushihui) on Unsplash*

Maybe it’s more of a crazy straw than a pipeline. It’s certainly more fun than useful at the moment.

I know, it’s 2020 and there are other CI/CD tools out there. Slicker tools. I don’t really know any of them. So, why Jenkins? Jenkins has been around, it supports a lot of build types and stages, and my unscientific research of querying the job listings on Indeed.com and LinkedIn.com show it to still be pretty well owning the market.

Additionally, I also have a project I’m working on currently that involves a react ui talking to a golang echo framework API. This toy project is something I know well and something that should build. This gave me a point of focus to explore with.

So, the first thing that I did was follow the Jenkins install instructions from the jenkins.io book. If you follow these instructions you’ll end up running Jenkins in Docker as part of a Docker-in-Docker or dind setup. This turns out to be less than ideal, but something I didn’t discover until I had implemented multiple stages and it was becoming more obvious that it was going to be a permanent pain point.

First issues also seen here:

  • SO
  • GitHub moby issue

Using the standard Jenkins agent for docker with a dind setup runs the build container “inside” the Jenkins container. But, as I and several others found, connecting to that inner container is non-obvious. I came across the “Do Not Use Docker-in-Docker for CI” post early, but I thought there must be a sensible way. As I learned a little more and reread the post I realized that there was not.

My recommendation is to follow jpetazzo’s advice when it comes to how you want Jenkins to build your code in a container. The only other issue I had was docker: Got permission denied while trying to connect to the Docker daemon socket at unix:///var/run/docker.sock so click the link and do the fix by Nashon paz.

Along the way I did actually get to see some green lights and play a while. Which was cool.

I almost immediately didn’t care for the Pipeline Builder Wizard and preferred to work directly with the Jenkinsfile. Maybe this will change, but I can’t imagine how at the moment. For being a wizard, the builder ui and options I found opaque as a newcomer. It’s easier for me to go stage by stage researching my options as I go.

I found this short video fairly helpful

Now I have a skeletal pipeline for the react piece. I think next I’ll try integrating some open source security tools. Stay tuned.
